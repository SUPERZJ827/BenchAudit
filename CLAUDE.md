# BenchCore — Claude 工作规范

## 项目概览

BenchCore 是一个 benchmark 质量缺陷检测 pipeline。核心代码在 `benchcore/`，实验结果在 `reports/`，数据集在 `/home/zhoujun/llmdata/datasets/`。

---

## 编码规范

### 1. 先读再写

修改文件前必须先读它，而不是只扫一眼。查看类似功能是怎么实现的（例如新增 auditor 前先看现有 auditor 的结构）。复用已有的工具函数，不要重新发明。检查文件顶部的 import，跟随项目已有的库选择。

### 2. 想清楚再动手

没搞清楚任务前不写代码。先说明假设和取舍。多种方案时给出推荐，最多列两三种。需求不清楚时停下来问，不要用"看起来合理"的代码补理解空白。

### 3. 保持简单

写能解决当前问题的最少代码。避免：
- 过早抽象（只有一个实现的接口/基类）
- 臆想式错误处理（处理根本不会发生的情况）
- 不必要的可配置（把永远不会变的值做成参数）

### 4. 外科手术式修改

diff 越小越好。不要碰用户没有要求修改的地方。匹配现有代码风格（引号、命名、格式）。不要顺手重构无关代码，不要重新格式化整个文件。每行改动都应能说出和任务直接相关的理由。

### 5. 验证

修 bug 时先写能复现 bug 的测试，看到失败，再修，再看到通过。改动前后都运行现有测试。测试关注行为，不关注实现细节。

运行测试：
```bash
python -m unittest discover tests/ -v
```

### 6. 目标驱动执行

多步骤任务执行前先列计划，让用户确认方向再开始。模糊需求转化为可验证的具体目标。

### 7. 调试

出问题时调查，不要猜。完整读错误信息和 stack trace。先复现再修。一次只改一处。卡住时说出已经试过什么、现象是什么、哪里不确定。

### 8. 依赖

不轻易添加新依赖。优先用项目已有的库（标准库 > 已有第三方 > 新引入）。必须添加时说明理由。

### 9. 沟通

说明做了什么、为什么这么做。主动指出隐患和不确定性。不解释用户已经知道的东西。精确表达不确定性（"我不确定 X 是否支持 Y"比"应该能工作"更有用）。

### 10. 常见失败模式（要避免）

- **大杂烩**：加一个功能顺手重构半个代码库
- **错误抽象**：为只出现一次的问题构建通用方案
- **隐形决策**：做了架构选择但没告诉用户
- **只有 happy path**：忽略 API 500、文件不存在、空输入等
- **知识幻觉**：使用不存在的 API 或幻想出来的库特性
- **风格漂移**：用自己偏好的风格而不是项目风格
- **失控重构**：修一个问题连锁扩散，忘了最初任务
